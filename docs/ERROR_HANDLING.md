# Error Handling

## 1. 기본 원칙

외부 서비스 하나의 실패가 전체 결과를 무조건 폐기하게 해서는 안 된다.

단, 핵심 데이터가 전혀 없거나 잘못된 결과를 발송할 위험이 있으면 안전하게 중단한다.

## 2. 오류 분류

### Retryable

일시적인 오류로 재시도할 수 있다.

- HTTP 429
- HTTP 500, 502, 503, 504
- 네트워크 타임아웃
- 일시적인 DNS 오류
- 외부 API 연결 실패

### Non-retryable

재시도해도 해결될 가능성이 낮다.

- API 키 누락
- 인증 실패
- 잘못된 데이터 스키마
- 잘못된 Notion Database ID
- 지원하지 않는 설정값
- 필수 기사 필드 누락

## 3. 단계별 정책

### Collect

한 수집처가 실패하면:

- 실패 내용을 `SourceCollectionReport`에 기록한다.
- 다른 수집처 수집은 계속한다.
- 전체 수집 기사가 0개이면 분석을 실행하지 않는다.

### Analyze

AI 응답 검증에 실패하면:

1. 원문 응답을 안전한 범위에서 로그로 기록한다.
2. 제한 횟수만 재시도한다.
3. 계속 실패하면 분석 단계를 실패 처리한다.
4. 검증되지 않은 결과를 발행하지 않는다.

### Publish

이메일 성공, Notion 실패처럼 일부 채널만 실패할 수 있다.

결과 상태:

- 모든 채널 성공: `success`
- 일부 채널 성공: `partial_success`
- 모든 채널 실패: `failed`

## 4. 재시도 정책

권장 기본값:

```text
최대 재시도: 2회
초기 대기: 1초
다음 대기: 2초
```

무한 재시도는 금지한다.

## 5. 타임아웃

모든 외부 요청은 타임아웃을 가져야 한다.

권장 기본값:

- RSS/HTTP: 10초
- AI API: 60초
- Notion API: 15초
- 이메일 API: 15초

Foundation 브랜치에서는 상수와 인터페이스만 준비할 수 있다.

## 6. 중복 실행

동일한 `targetDate`에 대해 이미 성공적으로 발행된 브리핑이 있으면 중복 발행하지 않아야 한다.

Foundation에서는 이를 위한 다음 개념을 타입으로 표현한다.

```ts
type PublicationDecision =
  | "publish"
  | "skip_already_published"
  | "retry_previous_failure";
```

실제 저장소 기반 중복 방지는 후속 브랜치에서 구현한다.

## 7. 로그 보안

로그에 기록하지 말아야 할 값:

- OpenAI API Key
- Notion API Key
- 이메일 API Key
- Gmail OAuth 토큰
- 사용자 비밀번호
- 전체 이메일 인증정보

기사 본문 전체도 기본 로그에 기록하지 않는다.

## 8. 사용자에게 잘못된 브리핑을 보내는 것보다 나은 선택

다음 상황에서는 발행을 중단한다.

- 분석 결과 JSON 검증 실패
- 출처 URL이 하나도 없는 뉴스
- 최종 뉴스가 0개
- 날짜가 목표 날짜와 맞지 않음
- 모든 수집처 실패
- AI가 필수 해설 필드를 반환하지 않음
