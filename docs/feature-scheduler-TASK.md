# feature/scheduler 기획서

# 1. 브랜치 개요

- **브랜치명:** `feature/scheduler`
- **목적:** Integration에서 완성된 경제 브리핑 파이프라인을 정해진 일정에 따라 자동 실행한다.
- **핵심 역할:** 실행 시점 관리와 자동화
- **실행 대상:** `feature/integration`에서 제공하는 전체 파이프라인 실행 진입점

이 브랜치는 뉴스 수집, AI 분석, Notion 저장 기능을 직접 수행하거나 다시 구현하지 않는다.

Scheduler는 오직 다음 질문에 책임을 가진다.

> 경제 브리핑 파이프라인을 언제, 어떤 기준으로, 얼마나 안전하게 자동 실행할 것인가?

---

# 2. Scheduler의 정확한 역할

전체 시스템은 다음과 같이 구분된다.

```text
News Collector
    뉴스 수집 방법을 책임진다.

AI Analysis
    수집된 뉴스 분석을 책임진다.

Notion Publisher
    분석 결과의 Notion 저장을 책임진다.

Integration
    위 기능들의 실행 순서와 연결을 책임진다.

Scheduler
    Integration 파이프라인의 자동 실행 시점을 책임진다.
```

즉, Integration이 다음 작업을 한 번 실행할 수 있도록 만든다면,

```text
전날 뉴스 수집
        ↓
AI 분석
        ↓
Notion 저장
```

Scheduler는 이 전체 작업을 정해진 시간마다 자동으로 시작한다.

```text
매일 정해진 시각
        ↓
Integration 파이프라인 호출
        ↓
전날 경제 브리핑 생성
```

---

# 3. 프로젝트 목표

사용자가 매일 직접 명령어를 실행하지 않아도 전날 경제·부동산·재테크 뉴스 브리핑이 자동으로 생성되어 Notion에 저장되어야 한다.

사용자가 기대하는 결과는 다음과 같다.

- 매일 정해진 시간에 자동 실행된다.
- 실행 날짜를 기준으로 전날 뉴스가 처리된다.
- 동일 날짜의 작업이 실수로 여러 번 중복 실행되지 않는다.
- 실행이 실패하면 실패 사실과 원인을 확인할 수 있다.
- 시스템이 잠시 중단되거나 늦게 시작되더라도 잘못된 날짜의 브리핑을 만들지 않는다.
- 개발자가 필요할 때 수동으로도 동일한 파이프라인을 실행할 수 있다.
- 자동 실행과 수동 실행이 서로 다른 결과를 만들지 않는다.

---

# 4. 핵심 의도

Scheduler의 목적은 단순히 일정마다 함수를 호출하는 것이 아니다.

다음 조건을 만족하는 **신뢰 가능한 일일 자동화**를 만드는 것이 목적이다.

1. 정확한 시간대에 실행된다.
2. 정확한 대상 날짜를 처리한다.
3. 같은 작업이 중복 실행되지 않는다.
4. 실행 상태와 결과를 추적할 수 있다.
5. 실패가 조용히 무시되지 않는다.
6. 외부 서비스 장애가 전체 시스템의 통제되지 않은 반복 실행으로 이어지지 않는다.
7. 운영 환경이 달라져도 시간 기준이 변하지 않는다.

---

# 5. 시간대 기준

모든 날짜와 실행 일정은 원칙적으로 다음 시간대를 기준으로 한다.

```text
Asia/Seoul
```

서버, CI 환경 또는 배포 환경이 UTC로 설정되어 있어도 실제 실행 기준은 서울 시간이어야 한다.

Scheduler는 서버의 로컬 시간 설정에 무조건 의존해서는 안 된다.

예를 들어 사용자가 매일 오전에 전날 브리핑을 받아보도록 설정했다면 다음 관계가 명확해야 한다.

```text
서울 기준 실행일: 2026-07-17
뉴스 수집 대상일: 2026-07-16
```

구체적인 자동 실행 시각은 운영 환경과 뉴스 수집 안정성을 고려하여 Claude가 제안한다.

단, 사용자가 브리핑을 확인하기 적절한 오전 시간대가 기본 방향이다.

---

# 6. 실행 대상

Scheduler가 직접 실행해야 하는 대상은 개별 모듈이 아니라 Integration에서 제공하는 전체 파이프라인이다.

올바른 흐름:

```text
Scheduler
    ↓
Integration
    ↓
News Collector
    ↓
AI Analysis
    ↓
Notion Publisher
```

잘못된 흐름:

```text
Scheduler
    ├── News Collector 직접 호출
    ├── AI Analysis 직접 호출
    └── Notion Publisher 직접 호출
```

Scheduler는 전체 실행 흐름을 다시 조립하지 않는다.

Integration이 제공하는 하나의 실행 단위를 호출해야 한다.

---

# 7. 자동 실행과 수동 실행

시스템은 자동 실행뿐 아니라 개발 및 장애 복구를 위한 수동 실행도 고려해야 한다.

## 7.1 자동 실행

정해진 일정에 따라 별도의 사용자 개입 없이 실행한다.

자동 실행 시 다음 정보가 명확해야 한다.

- 실제 실행 시각
- 기준 시간대
- 처리 대상 날짜
- 실행 식별 정보
- 실행 결과

## 7.2 수동 실행

개발자 또는 운영자가 필요할 때 파이프라인을 직접 실행할 수 있어야 한다.

예시 상황:

- 최초 환경 설정 검증
- 자동 실행 실패 후 재시도
- 특정 날짜 브리핑 복구
- 배포 후 동작 확인
- 테스트 환경 실행

자동 실행과 수동 실행은 가능한 한 동일한 Integration 진입점을 사용해야 한다.

Scheduler 때문에 수동 실행 결과와 자동 실행 결과가 달라져서는 안 된다.

---

# 8. 대상 날짜 정책

Scheduler는 “현재 시각에 실행한다”는 사실과 “어느 날짜의 뉴스를 처리한다”는 사실을 구분해야 한다.

기본 대상은 서울 기준 실행일의 전날이다.

```text
기본 처리 대상 날짜 = Asia/Seoul 기준 오늘 - 1일
```

다만 수동 복구 실행 등에서는 특정 대상 날짜를 명시할 필요가 있을 수 있다.

구체적인 입력 방식은 Claude가 설계하되 다음 요구사항을 만족해야 한다.

- 기본 실행은 전날을 처리한다.
- 필요 시 특정 날짜를 명확히 지정할 수 있다.
- 실행 지연 때문에 의도하지 않은 날짜가 처리되지 않는다.
- 실행 도중 날짜가 바뀌어도 처리 대상 날짜가 변경되지 않는다.
- 처리 대상 날짜는 실행 시작 시점에 확정되어 전체 파이프라인에서 동일하게 사용된다.

---

# 9. 중복 실행 방지

같은 날짜의 브리핑이 의도치 않게 여러 번 생성되지 않아야 한다.

중복 실행이 발생할 수 있는 상황은 다음과 같다.

- Scheduler가 같은 시각에 두 번 작동함
- 이전 실행이 끝나기 전에 다음 실행이 시작됨
- 서버가 재시작되면서 일정이 중복 등록됨
- 자동 실행 중 사용자가 수동 실행함
- 외부 실행 환경에서 동일 작업을 여러 번 호출함
- 실패한 것으로 오해하여 재실행했지만 실제 Notion 저장은 성공함

Scheduler는 이러한 상황을 고려해야 한다.

구체적인 중복 방지 방식은 Claude가 프로젝트 환경에 맞게 결정한다.

다만 최종적으로 다음을 구분할 수 있어야 한다.

- 최초 정상 실행
- 이미 완료된 날짜의 재실행 요청
- 현재 실행 중인 날짜의 중복 요청
- 실패한 작업에 대한 명시적 재실행
- 사용자가 의도적으로 다시 발행하려는 실행

중복 방지를 위해 무조건 모든 재실행을 금지해서는 안 된다.

장애 복구를 위한 명시적인 재실행 가능성은 유지해야 한다.

---

# 10. 동시 실행 정책

동일한 파이프라인이 동시에 여러 번 실행되어서는 안 된다.

특히 다음 문제가 발생하지 않아야 한다.

- 동일 뉴스에 대해 OpenAI API가 중복 호출됨
- 동일 날짜의 Notion 페이지가 여러 개 생성됨
- 한 실행의 로그와 다른 실행의 로그가 섞임
- 외부 API 사용량이 불필요하게 증가함
- 어느 실행이 최종 결과인지 알 수 없게 됨

이전 작업이 아직 실행 중일 때 새로운 실행 요청이 들어오면 어떤 정책을 적용할지 Claude가 제안한다.

가능한 정책의 예시는 다음과 같지만 구현 방법을 지정하지는 않는다.

- 새로운 실행을 건너뜀
- 대기 처리
- 명시적인 중복 오류 반환
- 일정 시간 이후 오래된 실행을 실패로 판단

선택한 정책과 이유를 작업 계획에서 설명해야 한다.

---

# 11. 실패 처리 원칙

Scheduler는 Integration의 실패를 성공으로 처리해서는 안 된다.

다음 실패를 구분할 수 있어야 한다.

- Scheduler 자체 실행 실패
- 설정 오류
- Integration 호출 실패
- 뉴스 수집 단계 실패
- AI 분석 단계 실패
- Notion 저장 단계 실패
- 실행 제한 시간 초과
- 예기치 않은 프로세스 종료

실패 시 최소한 다음 정보가 남아야 한다.

- 실행 대상 날짜
- 실행 시작 시각
- 실패 시각
- 실패한 단계
- 실패 원인
- 재실행 가능 여부
- 실행 식별 정보

Scheduler가 각 단계의 내부 오류를 임의로 바꾸거나 숨겨서는 안 된다.

Integration에서 제공한 오류 정보를 유지하면서 자동 실행 관점의 정보를 추가해야 한다.

---

# 12. 재시도 원칙

외부 API나 네트워크는 일시적으로 실패할 수 있다.

따라서 재시도 가능성을 고려하되, 무조건 반복해서 실행해서는 안 된다.

특히 다음 문제가 발생하지 않아야 한다.

- OpenAI API가 실패할 때 무한 반복
- Notion 저장 성공 후 응답 오류 때문에 페이지 중복 생성
- 잘못된 인증 정보로 계속 재시도
- 뉴스가 없는 정상 상황을 장애로 보고 반복
- 재시도 때문에 API 비용과 호출량이 과도하게 증가

어떤 오류를 재시도하고 어떤 오류는 즉시 실패로 처리할지 Claude가 제안한다.

재시도가 적용된다면 다음 원칙을 만족해야 한다.

- 횟수가 제한되어야 한다.
- 실패 원인이 기록되어야 한다.
- 동일 결과의 중복 발행 위험을 고려해야 한다.
- 인증·권한·설정 오류처럼 반복해도 해결되지 않는 오류는 구분해야 한다.
- 최종 실패 여부를 확인할 수 있어야 한다.

구체적인 재시도 구현은 Integration 또는 개별 모듈의 기존 책임과 중복되지 않도록 설계해야 한다.

---

# 13. 실행 지연 및 누락 정책

정해진 시각에 시스템이 실행되지 못할 수 있다.

예시:

- 서버가 꺼져 있었음
- 배포 중이었음
- 실행 환경 장애
- 네트워크 장애
- 일정 서비스 장애

이 경우 어떤 동작을 할지 명확한 정책이 필요하다.

다음 상황을 고려해야 한다.

## 13.1 실행 시각보다 늦게 시작된 경우

같은 날 조금 늦게 실행되더라도 전날 브리핑을 정확히 생성할 수 있어야 한다.

## 13.2 하루 전체가 누락된 경우

시스템이 다음 날 복구되었을 때 누락된 날짜의 브리핑을 자동으로 보충할지, 운영자가 수동으로 실행해야 할지 Claude가 비교하여 제안한다.

## 13.3 여러 날짜가 누락된 경우

한 번의 복구 과정에서 여러 날짜를 무제한으로 동시에 처리하여 외부 API를 과도하게 호출해서는 안 된다.

누락 복구 정책은 데이터 중복, API 비용, 운영 복잡도를 함께 고려해야 한다.

---

# 14. 실행 제한 시간

전체 파이프라인이 무기한 실행 상태로 남아 있어서는 안 된다.

뉴스 수집, AI 분석, Notion 저장 과정에는 외부 서비스 호출이 포함될 수 있으므로 비정상적으로 긴 실행을 감지할 수 있어야 한다.

실행 제한 시간이 초과된 경우 다음을 확인할 수 있어야 한다.

- 어느 단계에서 오래 걸렸는가
- 실행이 실제로 종료되었는가
- 재실행해도 안전한가
- 일부 결과가 이미 저장되었는가

구체적인 제한 시간과 처리 방식은 Claude가 실제 파이프라인 특성을 확인한 뒤 제안한다.

---

# 15. 설정 관리

Scheduler 실행에 필요한 설정은 코드에 직접 고정하지 않는다.

고려해야 할 설정 예시:

- 기준 시간대
- 자동 실행 여부
- 자동 실행 시각
- 실행 환경
- 제한 시간
- 재시도 관련 값
- 수동 실행 시 대상 날짜

실제 설정 항목과 관리 방식은 Claude가 프로젝트의 기존 구성 방식을 따른다.

민감정보를 로그나 코드에 노출해서는 안 된다.

Scheduler는 OpenAI 또는 Notion 인증정보를 직접 해석할 필요는 없지만, Integration 실행에 필요한 환경이 준비되어 있는지 확인할 수 있어야 한다.

---

# 16. 실행 환경 독립성

Scheduler가 어떤 환경에서 운영될지는 아직 변경될 수 있다.

예시:

- GitHub Actions
- 서버의 Cron
- 클라우드 Scheduler
- 상시 실행되는 Node.js 프로세스
- 별도 배치 실행 환경

이 기획서는 특정 실행 기술을 강제하지 않는다.

Claude는 현재 프로젝트와 예상 운영 환경을 분석하여 가장 적절한 방식을 제안한다.

다만 다음 조건은 반드시 충족해야 한다.

- Asia/Seoul 시간대가 정확해야 한다.
- 비밀값이 안전하게 관리되어야 한다.
- 실행 로그를 확인할 수 있어야 한다.
- 실패 여부를 알 수 있어야 한다.
- 동일 작업 중복 실행을 통제할 수 있어야 한다.
- 수동 실행과 장애 복구가 가능해야 한다.
- 무료 또는 합리적인 운영 비용을 우선 고려한다.

# 17. 관찰 가능성 및 실행 기록

자동화는 사람이 직접 보고 실행하지 않기 때문에 실행 상태를 확인할 수 있어야 한다.

최소 다음 내용을 추적할 수 있어야 한다.

- 실행 ID
- 실행 방식: 자동 또는 수동
- 처리 대상 날짜
- 기준 시간대
- 예정 실행 시각
- 실제 시작 시각
- 종료 시각
- 전체 소요 시간
- 성공 또는 실패
- 최종 완료 단계
- 실패한 단계
- 오류 정보
- 재시도 횟수
- 중복으로 건너뛴 실행 여부

로그 형식과 저장 방법은 Foundation 및 Integration의 공통 계약을 우선 사용한다.

Scheduler만의 별도 기록이 필요하다면 기존 구조와 중복되지 않도록 한다.

# 18. 알림과 운영 확인

최종적으로 자동 실행 실패를 사용자가 알 수 있어야 한다.

다만 이 브랜치에서 이메일 발송 기능이나 별도의 알림 서비스를 직접 구현할지는 프로젝트 범위와 기존 Publisher 구조를 확인한 뒤 판단한다.

최소한 실패가 로그에만 묻혀 사용자가 영원히 모르는 상태는 피해야 한다.

Claude는 다음을 구분하여 제안한다.

- 현재 브랜치에서 반드시 필요한 실패 확인 수단
- 추후 별도 알림 기능으로 분리할 수 있는 부분
- 운영 환경 자체가 제공하는 알림 기능 활용 가능성

불필요하게 새로운 알림 시스템을 크게 구현하지 않는다.

# 19. 정상 실행의 기대 결과

정상 실행 시 다음 흐름이 성립해야 한다.

```text
1. Scheduler가 정해진 시간에 실행된다.
2. 서울 기준 처리 대상 날짜를 확정한다.
3. 동일 날짜 실행 여부를 확인한다.
4. Integration 파이프라인을 한 번 호출한다.
5. Integration이 뉴스 수집, AI 분석, Notion 저장을 수행한다.
6. Scheduler가 최종 실행 결과를 기록한다.
7. 동일 날짜에 불필요한 중복 브리핑이 생성되지 않는다.
```

사용자는 별도의 수동 작업 없이 Notion에서 전날 브리핑을 확인할 수 있어야 한다.

# 20. 뉴스가 없는 경우

수집 결과가 0건인 경우를 무조건 Scheduler 장애로 판단하지 않는다.

다음 상황은 구분되어야 한다.

- 실제로 조건에 맞는 뉴스가 없음
- 뉴스 수집기가 장애로 기사를 가져오지 못함
- 날짜 계산 오류로 모든 기사가 제외됨
- 출처 전체가 실패함

이 판단의 구체적인 책임은 Integration과 News Collector의 결과 계약을 따른다.

Scheduler는 해당 결과를 임의로 재해석하지 않고 자동 실행 성공·실패 정책에 반영한다.

# 21. 멱등성 요구사항

같은 대상 날짜로 동일 작업을 여러 번 요청하더라도 시스템 상태가 통제 가능해야 한다.

특히 다음을 고려한다.

- 첫 실행이 완전히 성공한 경우
- AI 분석까지 성공하고 Notion 저장만 실패한 경우
- Notion 저장은 성공했지만 성공 응답을 받지 못한 경우
- 실행 도중 프로세스가 종료된 경우
- 운영자가 동일 날짜를 수동 재실행한 경우

Scheduler는 중복 방지 책임을 가지되, Notion Publisher와 Integration의 중복 처리 정책을 존중해야 한다.

모든 멱등성 책임을 Scheduler 한 곳에 억지로 몰아넣지 않는다.

각 계층의 책임 범위를 Claude가 명확히 정리해야 한다.

# 22. 테스트 관점

구체적인 테스트 방식과 도구는 Claude가 결정한다.

다만 다음 시나리오는 반드시 검증되어야 한다.

## 22.1 시간 및 날짜

- Asia/Seoul 기준으로 일정이 해석되는가
- UTC 환경에서도 올바른 서울 날짜를 사용하는가
- 실행일의 전날이 처리 대상이 되는가
- 자정 전후에도 대상 날짜가 흔들리지 않는가
- 월말, 연말, 윤년 날짜를 정확히 처리하는가
- 실행 시작 후 날짜가 변경되어도 대상 날짜가 유지되는가

## 22.2 정상 실행

- 정해진 시점에 Integration이 한 번 호출되는가
- Integration 결과가 Scheduler 실행 결과에 반영되는가
- 자동 실행과 수동 실행이 같은 파이프라인을 사용하는가
- 성공 상태와 완료 정보가 기록되는가

## 22.3 중복 및 동시 실행

- 같은 날짜의 중복 자동 실행을 방지하는가
- 자동 실행과 수동 실행이 겹치는 경우를 통제하는가
- 이전 작업이 실행 중일 때 새로운 작업을 통제하는가
- 명시적인 복구 재실행은 가능하게 할 수 있는가

## 22.4 실패

- Integration 호출 실패를 기록하는가
- 단계별 실패 정보를 유지하는가
- 타임아웃을 감지하는가
- 설정 오류를 명확히 반환하는가
- 인증·권한 오류를 무한 재시도하지 않는가
- 최종 실패를 성공으로 표시하지 않는가

## 22.5 재시도

- 재시도 가능한 오류만 제한적으로 재시도하는가
- 재시도 횟수가 제한되는가
- 재시도 과정이 기록되는가
- 성공 이후 중복 발행으로 이어지지 않는가

## 22.6 실행 지연 및 복구

- 예정 시각보다 늦게 실행되어도 정확한 날짜를 처리하는가
- 누락된 날짜의 복구 정책이 의도대로 동작하는가
- 여러 날짜를 무제한으로 동시에 처리하지 않는가

## 22.7 계약

- Foundation의 공통 타입과 오류 계약을 준수하는가
- Integration 진입점을 통해서만 전체 파이프라인을 실행하는가
- 개별 모듈을 Scheduler에서 재구현하거나 직접 조립하지 않는가

# 23. Foundation 및 Integration과의 관계

## 23.1 Foundation

Foundation에서 제공하는 다음 요소를 우선 확인한다.

- 공통 설정
- 실행 결과 타입
- ExecutionLog
- AppError
- ErrorCodes
- 시간대 및 날짜 유틸리티
- 타임아웃 및 재시도 정책
- Result 관련 계약

공통 계약은 승인 없이 변경하지 않는다.

변경이 필요하다고 판단되면 먼저 다음을 보고한다.

1. 현재 계약으로 구현이 어려운 이유
2. 변경이 필요한 항목
3. 다른 브랜치에 미치는 영향
4. 하위 호환성 유지 방안
5. 계약 변경 없이 해결 가능한 대안

## 23.2 Integration

Scheduler는 Integration에서 제공하는 전체 파이프라인의 공개 실행 진입점만 사용한다.

Integration 내부 로직을 복제하지 않는다.

Integration에 적절한 공개 실행 진입점이 없다면 임의로 우회하지 말고 다음을 먼저 보고한다.

- 현재 Integration 구조
- Scheduler에서 필요한 계약
- 최소 변경 범위
- 다른 모듈에 미치는 영향
- 대체 방법

위 값은 예시이며 실제 완료 보고에서는 실제 실행 결과와 설정을 사용한다.

# 핵심 원칙

> Integration은 경제 브리핑을 한 번 생성하는 전체 실행 흐름을 책임진다.

> Scheduler는 그 Integration 파이프라인을 언제 자동으로 실행할지 책임진다.

> Scheduler는 뉴스 수집, AI 분석, Notion 저장을 직접 구현하지 않는다.

> 서울 기준 대상 날짜, 중복 실행 방지, 실패 추적, 수동 복구까지 고려하여 사용자가 매일 안정적으로 브리핑을 받을 수 있도록 만드는 것이 이 브랜치의 목적이다.

> 구현 방식은 Claude가 선택하되, Foundation과 Integration의 공통 계약 및 본 기획서의 운영 요구사항을 준수해야 한다.
