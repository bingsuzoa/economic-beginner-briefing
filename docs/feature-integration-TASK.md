# feature/integration 기획서

# 1. 브랜치 개요

- **브랜치명:** `feature/integration`
- **목적:** 각 브랜치에서 독립적으로 구현된 기능을 하나의 실행 가능한 파이프라인으로 연결한다.
- **역할:** 시스템의 전체 흐름을 완성하는 오케스트레이션(Orchestration)

이 브랜치는 새로운 기능을 만드는 곳이 아니라, 이미 구현된 기능들이 하나의 서비스처럼 동작하도록 연결하는 것이 목적이다.

---

# 2. 프로젝트 목표

사용자는 "경제 브리핑 생성" 버튼을 누르거나 자동 실행이 시작되면 하나의 과정만 보게 된다.

내부적으로는 다음과 같은 단계가 순차적으로 수행되어야 한다.

```
전날 뉴스 수집
        ↓
AI 분석
        ↓
브리핑 생성
        ↓
Notion 저장
```

사용자는 각 모듈의 존재를 알 필요가 없으며, 하나의 서비스처럼 동작해야 한다.

---

# 3. 브랜치 책임

이 브랜치가 책임지는 것은 다음과 같다.

- 각 모듈 연결
- 전체 실행 순서 관리
- 데이터 전달
- 단계별 성공/실패 관리
- 전체 실행 결과 반환
- 공통 계약 준수 여부 확인
- 전체 파이프라인 검증

---

# 4. 책임지지 않는 범위

다음 기능은 구현하지 않는다.

- 뉴스 수집 로직
- AI 분석 로직
- Notion 저장 로직
- 이메일 발송
- 스케줄링
- UI 개발

각 기능은 해당 브랜치에서 구현되어야 하며 Integration은 재구현하지 않는다.

---

# 5. 전체 파이프라인 목표

최종적으로 다음 흐름이 하나의 작업처럼 수행되어야 한다.

```
News Collector
      ↓
Article[]
      ↓
AI Analysis
      ↓
Briefing
      ↓
Notion Publisher
      ↓
Execution Result
```

Integration은 데이터가 각 단계를 올바르게 통과하도록 보장해야 한다.

---

# 6. 데이터 계약

각 모듈은 Foundation에서 정의한 공통 계약을 그대로 사용해야 한다.

Integration은 모듈 사이에서 불필요한 데이터 변환을 최소화해야 한다.

공통 타입은 승인 없이 변경하지 않는다.

---

# 7. 실행 순서

실행 순서는 항상 논리적으로 유지되어야 한다.

예를 들어

- 뉴스가 준비되지 않았는데 AI 분석을 시작해서는 안 된다.
- AI 분석이 실패했는데 Notion 저장을 시도해서는 안 된다.

실행 순서와 조건은 Claude가 가장 적절한 방식으로 설계한다.

---

# 8. 실패 정책

전체 서비스는 한 단계의 실패 원인을 명확히 확인할 수 있어야 한다.

다음과 같은 상황을 구분해야 한다.

- 뉴스 수집 실패
- 뉴스가 없음
- AI 분석 실패
- Notion 저장 실패
- 예기치 않은 시스템 오류

실패는 조용히 무시되지 않아야 한다.

후속 단계가 실행 가능한지 여부도 명확해야 한다.

---

# 9. 실행 결과

최종 실행 결과에서는 최소 다음 정보를 확인할 수 있어야 한다.

- 실행 성공 여부
- 실행 시작 시간
- 실행 종료 시간
- 수행된 단계
- 실패한 단계
- 실패 원인
- 최종 생성 결과

실행 과정은 추후 운영 및 디버깅에 활용할 수 있어야 한다.

---

# 10. 데이터 무결성

Integration은 데이터를 변경하거나 새롭게 생성하지 않는다.

각 모듈의 출력이 다음 모듈의 입력으로 안전하게 전달되는 것이 가장 중요하다.

데이터 의미가 변경되어서는 안 된다.

---

# 11. 테스트 관점

반드시 검증되어야 하는 사항

- 전체 파이프라인 정상 실행
- 단계별 데이터 전달
- 단계별 실패 처리
- 뉴스가 없는 경우
- AI 실패 시 동작
- Notion 실패 시 동작
- 실행 결과 반환
- Foundation 계약 준수

테스트 방식은 Claude가 결정한다.

---

# 12. Foundation 계약

반드시 Foundation의 공통 계약을 따른다.

예시

- NewsCollector
- NewsAnalyzer
- BriefingPublisher
- ExecutionLog
- Result
- AppError
- ErrorCodes

공통 계약은 승인 없이 변경하지 않는다.

---

# 13. 각 브랜치와의 관계

Integration은 아래 브랜치들의 구현을 연결한다.

- feature/news-collector
- feature/ai-analysis
- feature/notion-publisher

각 브랜치는 독립적으로 개발되어야 하며 Integration은 이를 조합하는 역할만 수행한다.

---

# 14. Claude 작업 지시

이 문서는 구현 방법을 지정하지 않는다.

Claude는 프로젝트를 분석한 후

- 전체 연결 방식
- 실행 흐름
- 오류 처리
- 실행 결과 관리
- 테스트 전략

을 프로젝트 구조에 맞게 설계한다.

작업 시작 전 반드시

1. CLAUDE.md 확인
2. DATA_CONTRACTS.md 확인
3. Foundation 계약 확인
4. 각 구현 브랜치의 인터페이스 확인
5. Mock Pipeline 확인

을 수행한다.

코드를 수정하기 전에 다음을 먼저 보고한다.

- 전체 연결 전략
- 실행 흐름
- 단계별 책임
- 실패 처리 전략
- 데이터 전달 전략
- 테스트 전략
- 예상 변경 파일
- 필요한 의존성
- 위험 요소
- 기획과 다른 점이 있다면 그 이유

승인 후 구현을 시작한다.

---

# 15. 완료 조건

- News Collector와 연결된다.
- AI Analysis와 연결된다.
- Notion Publisher와 연결된다.
- 전체 파이프라인이 정상 수행된다.
- 단계별 실패를 올바르게 처리한다.
- 실행 결과를 제공한다.
- Foundation 계약을 준수한다.
- 테스트 통과
- TypeScript 통과
- Lint 통과
- Build 통과

---

# 핵심 원칙

> Integration은 기능을 새로 만드는 브랜치가 아니다.

> 이미 구현된 모듈들을 하나의 서비스처럼 연결하여, 사용자가 "경제 브리핑 생성"이라는 하나의 기능으로 사용할 수 있도록 만드는 것이 목적이다.

> 구현 방식은 Claude가 자유롭게 선택하되, Foundation의 공통 계약과 본 기획서의 요구사항을 반드시 만족해야 한다.
