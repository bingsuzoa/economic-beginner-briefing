# feature/notion-publisher 기획서

# 1. 브랜치 개요

- **브랜치명:** `feature/notion-publisher`
- **목적:** AI Analysis가 생성한 분석 결과를 사람이 읽기 좋은 형태로 Notion에 기록한다.
- **입력:** Foundation에서 정의한 AI 분석 결과(Briefing 및 관련 계약)
- **출력:** 사용자의 Notion Workspace에 생성된 브리핑 페이지

이 브랜치는 **분석 결과를 Notion으로 전달하는 역할만 담당**한다.

뉴스 수집, AI 분석, 이메일 발송, 스케줄링은 이 브랜치의 책임이 아니다.

---

# 2. 프로젝트 목표

매일 생성되는 경제 브리핑을 사용자가 Notion에서 하나의 지식 자산으로 관리할 수 있도록 한다.

단순히 텍스트를 저장하는 것이 아니라,

- 읽기 쉽고
- 구조화되어 있으며
- 나중에 다시 찾아보기 쉽고
- 하루 단위 기록이 누적되는

브리핑 저장소를 만드는 것이 목적이다.

---

# 3. 사용자 관점

사용자는 매일 Notion을 열면 다음과 같은 흐름을 기대한다.

```
2026-07-16 경제 브리핑

오늘의 핵심 요약

주요 뉴스

왜 중요한가

생활에 미치는 영향

앞으로 지켜볼 내용
```

Notion은 단순 저장 공간이 아니라
브리핑 아카이브 역할을 해야 한다.

---

# 4. 브랜치 책임

이 브랜치가 책임지는 범위

- AI 분석 결과 수신
- Notion에 적합한 문서 구조 생성
- 보기 좋은 레이아웃 구성
- 페이지 생성 또는 저장
- 저장 성공 여부 반환
- 실패 시 오류 정보 반환

---

# 5. 책임지지 않는 범위

다음은 구현하지 않는다.

- 뉴스 수집
- OpenAI 호출
- AI 분석
- 기사 요약 생성
- 이메일 발송
- GitHub Actions
- 데이터베이스
- 사용자 화면

---

# 6. 저장 목적

Notion에는 하루에 하나의 브리핑이 저장되는 것을 목표로 한다.

사용자는 날짜별 브리핑을 누적하여

- 경제 흐름
- 정책 변화
- 부동산 변화
- 금융시장 변화

를 시간순으로 확인할 수 있어야 한다.

---

# 7. 문서 품질

Notion 문서는

- 읽기 쉽고
- 제목과 본문이 명확히 구분되고
- 너무 긴 텍스트 덩어리가 없으며
- 중요 내용이 눈에 띄도록 구성되어야 한다.

단순 JSON 출력처럼 저장되어서는 안 된다.

---

# 8. 저장 정책

브리핑 한 건은 하나의 독립적인 페이지가 된다.

하루 브리핑이 이미 존재하는 경우의 처리 정책은 Claude가 제안한다.

예를 들어

- 덮어쓰기
- 업데이트
- 중복 생성 방지
- 새 버전 생성

등의 전략을 비교하여 가장 적절한 방식을 선택한다.

---

# 9. 데이터 정확성

Notion에 저장되는 내용은

AI Analysis가 생성한 결과와 의미가 달라져서는 안 된다.

Publisher는

- 표현을 보기 좋게 정리할 수는 있지만
- 내용을 새롭게 생성하거나
- 의미를 바꾸거나
- 임의로 삭제해서는 안 된다.

---

# 10. 장애 처리

Notion 저장 실패가 발생하더라도

실패 원인을 확인할 수 있어야 한다.

다음 상황을 고려한다.

- 인증 실패
- 권한 부족
- API 오류
- 네트워크 오류
- 요청 제한
- 잘못된 데이터

실패 정보는 Foundation 계약에 맞게 반환한다.

---

# 11. 실행 결과

최소 다음 정보를 확인할 수 있어야 한다.

- 저장 성공 여부
- 생성된 페이지 정보
- 실행 시간
- 실패 원인
- 재시도 가능 여부

---

# 12. 테스트 관점

반드시 검증해야 하는 사항

- 정상 저장
- 긴 브리핑 저장
- 빈 값 처리
- 특수문자 처리
- 한글 정상 저장
- 날짜별 페이지 생성
- 중복 저장 정책
- API 실패 처리
- 권한 오류 처리
- 타입 계약 준수

테스트 방법과 구현 방식은 Claude가 결정한다.

---

# 13. Foundation 계약

반드시 Foundation에서 정의한

- Briefing
- BriefingPublisher
- ExecutionLog
- AppError
- ErrorCodes

등 공통 계약을 따른다.

공통 타입은 승인 없이 변경하지 않는다.

---

# 14. AI Analysis와의 관계

Publisher는

AI Analysis가 생성한 결과를 그대로 입력으로 사용한다.

Publisher는

AI가 어떻게 분석했는지 알 필요가 없으며

분석 로직을 포함해서도 안 된다.

---

# 15. Claude 작업 지시

이 문서는 구현 방법을 지정하지 않는다.

Claude는 프로젝트를 분석한 후

- Notion 저장 방식
- 페이지 구성
- API 사용 방식
- 오류 처리
- 테스트 전략

을 스스로 설계한다.

작업 시작 전 반드시

1. CLAUDE.md
2. DATA_CONTRACTS.md
3. Foundation 코드
4. AI Analysis 계약
5. Mock Publisher

를 확인한다.

코드를 수정하기 전에 아래 내용을 먼저 보고한다.

- Notion 저장 전략
- 페이지 구성 전략
- 중복 저장 전략
- 오류 처리 전략
- 테스트 전략
- 예상 변경 파일
- 필요한 라이브러리
- 위험 요소
- 기획과 다른 점이 있다면 그 이유

승인 후 구현을 시작한다.

---

# 16. 완료 조건

다음을 모두 만족하면 완료로 본다.

- AI Analysis 결과를 입력으로 받을 수 있다.
- Notion에 정상 저장된다.
- 사람이 읽기 좋은 구조를 제공한다.
- 오류 처리가 가능하다.
- Foundation 계약을 준수한다.
- 테스트 통과
- TypeScript 통과
- Lint 통과
- Build 통과

---

# 핵심 원칙

> 이 브랜치의 목적은 Notion API를 호출하는 것이 아니다.

> AI가 생성한 경제 브리핑을 사용자가 매일 읽고, 장기적으로 축적하며, 다시 찾아볼 수 있는 형태로 안전하게 보관하는 것이 목적이다.

> 구현 방식은 Claude가 자유롭게 선택하되 Foundation 계약과 본 기획서의 요구사항을 만족해야 한다.
